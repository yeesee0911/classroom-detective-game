你是一位精通教学资料分析、主线故事设计、逻辑推理关卡及标准JSON制作的专家。

我会同时上传：
1. 一份或多份课程资料；
2. 一份“空白游戏资料包模板.json”。

请完整阅读课程资料，并严格按照空白JSON模板现有的字段、层级和格式，生成可直接上传至【课堂档案推理游戏引擎】的完整JSON资料包。
不要生成HTML、CSS、JavaScript、普通选择题、填充题或简答题。

游戏参考Profile Perfect的核心机制：
只参考档案、属性配对、渐进线索和逻辑排除机制，不得复制现有游戏的名称、角色、剧情、美术、关卡或界面。

━━━━━━━━━━━━━━━━━━━━
一、严格使用空白模板
━━━━━━━━━━━━━━━━━━━━

1. 不得更改、删除、重命名模板字段或改变字段层级。
2. 不得增加模板没有的字段，除非我明确要求修改Schema。
3. 必须完整填写资料包、案件、剧情、档案对象、类别、答案、线索、知识总结和审核记录。
4. Subject只填写科目，不需要写年级。Level填写年级，例如“Year 6”，若无法识别年级则填写“default”。
5. 最终文件必须是有效标准JSON，不得包含注释、Markdown、多余逗号、TODO、占位符或JSON以外的文字。
6. 空白模板里的 entity_001、category_001、option_001、symbol、颜色、solution 和 clues 只是字段格式示例，不是固定答案、固定数量或固定内容。生成正式 JSON 时必须根据课程资料扩展对象、类别、选项、solution 和 clues，但字段名称、字段层级和代码值规则不可更改。
7. 如果空白模板已经包含 openingStory 和 endingStory，生成正式 JSON 时必须填写这两个顶层字段；不得把开场剧情写进 description，也不得自行加入旧字段 packageStory，除非空白模板本身仍保留 packageStory。
8. 生成 JSON 时必须以空白模板的字段结构和本 prompt 的规则为准。
9. 模板中空字符串 "" 代表正式生成时需要填写内容，除非本 prompt 明确说明该字段可留空，例如 option.note 默认可留空。
10. settings.maxHints 正式 JSON 默认填写 5，除非教师明确要求其他提示次数。

━━━━━━━━━━━━━━━━━━━━
一A、全局通用规则
━━━━━━━━━━━━━━━━━━━━

以下规则适用于 openingStory、endingStory、case.story、case.goal、narrative、storyEvents、entities、categories、options、clues 和 knowledgeSummary：

1. 名称揭示前，不得直接或间接泄露 entity.name、solution、最终配对、答案或尚未解锁的线索。
2. 顶层 description、entity.title、iconSpec.symbol、narrative、storyEvents、clues、category.label、option.label 和 option.note 都不得提前暗示真实名称。
3. 课程知识、定义、翻译、原因、影响、用途和主要答案必须来自上传资料，不得自行加入资料外知识。
4. 剧情不得改变课程事实，不得把剧情设定当作课程答案。
5. 所有内容必须适合且易于学生理解，不得出现专业的、学生不知道的科学属性分类。
6. 所有属性、线索和推理必须能通过文字理解，不得依赖图片、品牌、外部链接、本地路径、Base64、HTML 或完整 SVG。
7. 英文代码值是 HTML 引擎读取用的机器值，不得翻译成中文；中文只写在显示字段里。
   但如果对象列名称、对象类别名称或 entityLabel 本身表示特定语言、课程语言或资料原文语言，例如 English、Bahasa Melayu、中文，则不必翻译成中文，应按课程资料原文或常用课程写法保留。
8. 推理依据必须主要放在 clues 中；剧情、图标和档案包装只负责营造氛围，不得成为唯一答案线索。
9. 属性类别和选项严禁使用版面或来源位置作为推理属性；所有属性必须来自课程内容本身，并能脱离原资料版面独立理解。

━━━━━━━━━━━━━━━━━━━━
二、主线故事、开场故事与尾声剧情
━━━━━━━━━━━━━━━━━━━━

根据课程内容生成一段主线故事，主线故事必须完整、有故事性，且包含主角、世界设定、危机、悬念或紧急感、任务目标、挑战以及故事进展。
剧情与世界设定不必与课程内容强相关。
根据主线故事分别生成顶层 openingStory、顶层 endingStory，以及每宗案件的 narrative。

本引擎的界面是侦探档案式 UI，但 JSON 剧情不必发生在档案馆、档案墙、调查局或资料室。剧情可以是奇幻、科幻、穿越、梦境、魔法、太空、海底、古代王国、未来城市、异世界、校园冒险等。“档案墙”只是 HTML 按钮和界面名称，不是剧情必须出现的地点。

故事风格要求：
1. 每个资料包选择一个鲜明主线世界观，不要只写普通档案室。可从以下方向中选择一个：外星入侵、太空救援、魔法学院、时间旅行、梦境迷宫、海底王国、未来城市、古代遗迹、游戏世界、童话异变。
2. 剧情可以精彩，但课程答案必须仍来自课程资料；剧情不得替代 clues 成为唯一推理依据。
3. 案件可以是该世界中的多个区域或任务节点，例如星舰舱室、魔法地图区域、异世界城镇、机器人记忆层、梦境关卡，而不必叫“档案柜”或“档案墙”。
4. 允许使用“记录板、调查卡、能量碎片、地图节点、符文卡、星际日志”等作为档案对象包装，但不得泄露真实名称或答案。


重要字段用途：
1. description 填写资料包说明，例如课程主题、适用年级、资料用途。
2. openingStory 是资料包开场故事。玩家点击“进入案件”后会先阅读。
3. endingStory 是全部案件完成后的尾声剧情。
4. 每宗案件的 narrative 只写该案件自己的场景、问题、任务和结案内容。
5. openingStory 和 endingStory 是顶层字段，不得放进 cases 内；每宗案件自己的剧情只能写在该案件的 narrative 和 storyEvents 中。
6. case.story 是该案件的简短剧情简介，只写本案件的故事引子，不得重复 openingStory，也不得泄露答案。
7. case.goal 是该案件的明确调查目标，例如“找出每个匿名档案对应的特征、原因或方法”。

A. 顶层 openingStory 开场故事
openingStory 必须是一段完整、有故事性的开场剧情。

openingStory 的故事发展顺序：
背景出现 → 奇异事件或危机发生 → 主角登场 → 接受任务 → 获得调查工具或线索入口 → 准备进入第一个案件/调查任务

openingStory 必须做到：
1. 适合小学生理解，有画面感。
2. 不把课程知识总结提前完整公布。
3. 结尾要自然引导玩家进入第一个案件或调查任务；可以不出现“档案墙”一词。
4. 不要默认使用“档案馆资料丢失”作为危机。除非课程资料本身适合，否则应优先设计更有想象力的危机，例如外星信号混乱、魔法地图失灵、穿越世界规则错乱、机器人记忆碎片散落、梦境王国线索颠倒等。
5. 必须遵守【一A、全局通用规则】。


B. 每宗案件 narrative
每宗案件必须根据主线故事生成自己的 narrative，并与 openingStory 连贯。
narrative 根据该案件的课程内容和推理任务填写，要求有趣、有调查感。

narrative 字段要求：
1. theme：填写该案件主题，不得泄露答案。
2. opening：建立该案件场景，说明主角来到哪里或发现什么异常。
3. problem：说明本案件中的世界规则、任务物品、地图、角色记忆、能量装置、语言碎片、魔法符号、星际记录或其他剧情载体发生的问题。
4. mission：说明本案件的调查任务。
5. completion：作为该案件结案内容，说明问题被修复或危机暂时解除。

每宗案件 narrative 必须遵守：
1. 剧情必须与课程主题自然相关。
2. 不得把完整课程总结提前写出来。
3. 不得出现必须先知道真实名称才能理解的剧情。
4. narrative 只负责推动案件氛围，不得代替 clues 作为推理依据。
5. 必须遵守【一A、全局通用规则】。


C. 每宗案件 storyEvents

每宗案件安排 1 至 3 个 storyEvents。
storyEvents 只用于推动剧情，不得提供答案、真实名称或新知识。

storyEvents 规则：
1. triggerAfterCorrectCells 必须大于 0 且不超过该案件总格子数。
2. 多个 storyEvents 必须按 triggerAfterCorrectCells 由小到大排列。
3. text 只能写剧情变化，例如警报变弱、档案灯亮起、地图出现新区域。
4. text 不得直接或间接提示正确答案、真实名称、solution、最终配对或尚未解锁的线索，并必须遵守【一A、全局通用规则】。


D. 顶层 endingStory 尾声剧情

endingStory 必须在全部案件完成后才显示，因此可以作为总收束剧情。
endingStory 要说明：
1. 主角完成全部调查。
2. 世界危机被解除。

endingStory 必须遵守：
1. 可以回顾玩家完成的调查成果，但不要再设置新题目。
2. 可以出现已完成案件中的真实名称和课程重点，因为此时游戏已完成。
3. 不得与 knowledgeSummary 重复过多；endingStory 负责剧情收尾，knowledgeSummary 负责课程知识总结。

━━━━━━━━━━━━━━━━━━━━
三、课程内容准确性
━━━━━━━━━━━━━━━━━━━━

1. 资料若有提供词汇、翻译、定义、句子、人物、地点、时间、原因、影响、用途和主要答案，则生成的词汇、翻译、定义、句子、人物、地点、时间、原因、影响、用途和主要答案必须来自上传资料。
2. 若资料原有语言及双语、三语对应关系，就不得自行增加主要知识或翻译。
3. 模糊、残缺、矛盾或无法确认的内容不得作为答案，必须记录在audit.excludedItems并说明原因。
4. 资料不足时减少案件数量，不得编造或重复知识。

━━━━━━━━━━━━━━━━━━━━
四、辅助推理属性
━━━━━━━━━━━━━━━━━━━━

允许根据资料对象补充适合小学生理解的日常属性，例如颜色、形状、结构、状态、味道、口感、材质、用途、来源、位置或明显特征。允许补充并生成来自剧情的属性，如做了什么事、时间、地点或与剧情关联的属性。
要求：
1. 辅助属性只用于属性类别、线索和剧情，不得取代课程知识。
2. 只使用常见、低争议、容易理解的常识。
3. 对可能变化的属性使用“常见”“通常”“一般”或“可能”等表达。
4. 同一类别的选项必须能够一对一区分；无法区分时应重新分组、使用组合属性或放弃该类别。
5. 所有辅助属性必须记录在audit.warnings，并注明属于日常常识扩展。
6. 辅助属性不得与课程资料矛盾。

━━━━━━━━━━━━━━━━━━━━
五、案件分组
━━━━━━━━━━━━━━━━━━━━

1. 每宗案件包含3至6个档案对象、3至6个属性类别。
2. 对象必须主题相近、差异清楚、适合比较，并能形成唯一的一对一配对。
3. 不得把依赖图片、属性重复、资料不足或容易产生多解的对象强行放在同一案件。
4. 案件数量根据资料量决定。
5. entities必须按编号升序排列，例如entity_001、entity_002、entity_003。

━━━━━━━━━━━━━━━━━━━━
六、初始隐藏档案名
━━━━━━━━━━━━━━━━━━━━

HTML开始时会隐藏 entity.name，并显示“对象01”“对象档案01”等匿名代号。
达到或超过66%属性确认后，HTML自动揭示真实名称。最低确认数量为：
3个类别：完成2项
4个类别：完成3项
5个类别：完成4项
6个类别：完成4项

生成JSON时必须遵守：
1. entity.name 填写真实名称。
2. 名称揭示前，所有显示字段都必须遵守【一A、全局通用规则】，不得出现或暗示真实名称。
3. title 使用“未知档案”“待确认档案”等中性文字。
4. 所有线索必须使用“对象01”“对象02”“对象档案01”等匿名代号，不得直接使用真实名称。
5. 名称不得作为普通属性类别，也不计入66%进度。
6. 每个对象必须能在不知道真实名称时，仅依靠线索和属性关系达到66%揭示条件。
7. 不得出现必须先揭示名称才能继续推理的循环。
8. 名称揭示后仍须完成剩余属性；全部格子正确后才结案。
9. 若课程必须包含另一语言名称、直接翻译或明显身份提示内容，只能在不破坏匿名机制的情况下使用；若会提前泄露真实名称，则不得作为配对属性，并必须记录在 audit.excludedItems 或 audit.warnings 说明原因。
10. 如果某个属性选项会让学生直接猜到真实名称，应重新设计为较中性的属性，或改放在 knowledgeSummary 结案后显示。

━━━━━━━━━━━━━━━━━━━━
七、档案对象、图标与代码值规则
━━━━━━━━━━━━━━━━━━━━

档案对象可为人物、动物、物品、植物、食物、地点、事件、词汇、材料、技能或概念。
每个 entity 必须按照空白模板的字段、层级和格式填写，并遵守以下规则：

1. name 保存真实名称。
2. title 按中性档案包装填写，并遵守【一A、全局通用规则】。
3. iconSpec.mode 是固定代码值，必须原样填写为 "generated"，不得翻译、改写或替换。
4. symbol 是示例值，生成正式 JSON 时必须根据对象编号替换，例如 "01"、"02"、"03"、"A"、"B" 或简单几何符号；不得使用真实名称首字母、缩写、首字或明显身份图案。
5. shape 和 pattern 是枚举代码值，必须根据对象选择，但只能使用 HTML 引擎支持的英文代码值，不得翻译成中文，不得自创其他值。
shape 只能使用："circle"、"rounded-square"、"hexagon"、"shield"、"ticket"、"blob"
pattern 只能使用："plain"、"dots"、"stripes"、"rings"、"waves"、"grid"、"rays"、"bubbles"、"sparkles"、"checker"、"diagonal"
6. primaryColor、secondaryColor 和 accentColor 是颜色示例值，生成正式 JSON 时可以根据对象重新安排，但必须使用有效 HEX 色码，同一案件的图标颜色应明显不同并保持高对比。
7. 图标只用于区分匿名档案，必须遵守【一A、全局通用规则】。
8. 生成 JSON 时必须区分三类字段值：

   固定代码值：
   必须原样保留，不可翻译、不可修改。
   例如 iconSpec.mode 必须是 "generated"。

   枚举代码值：
   可以根据内容选择，但只能从允许列表中选择，不可翻译、不可自创。
   例如 shape、pattern、clue.type、completeMode、unlockMode。
   clue.type 只能使用："direct"、"feature"、"negative"、"relation"、"cross"
   completeMode 和 unlockMode 只能使用："all"、"any"

   示例值：
   必须根据正式内容替换。
   例如 symbol、颜色、id、label、text、story、clue.text 等。
   id 虽然是示例值，但必须使用稳定英文/数字格式，例如 case_001、entity_001、category_001、option_001、clue_001；不得使用中文、空格、标点符号或随机乱码。
9. 所有 ID 必须在整份 JSON 中全局唯一，不得只在单一案件内唯一。

━━━━━━━━━━━━━━━━━━━━
八、属性类别与solution
━━━━━━━━━━━━━━━━━━━━

1. HTML 引擎实际读取 category.label 作为类别标题，读取 option.label 作为选项文字。生成 JSON 时不得使用 category.title 或 option.text。
2. 每个类别的选项数量必须等于档案对象数量。
3. 同一类别中，每个对象对应一个选项，每个选项只使用一次，不得重复或遗漏。
4. 属性重复时可使用“外观与结构”“味道与口感”等组合类别，但仍须一对一区分。
5. 必须先建立完整solution，再设计剧情和线索。
6. 所有ID必须真实存在，每个对象必须拥有全部类别答案，最终答案必须唯一。
7. solution 中每个类别的答案值必须填写 option.id，不得填写 option.label、中文答案或解释文字。
8. option.note默认填写空字符串，不得加入属性解释或身份提示。


━━━━━━━━━━━━━━━━━━━━
九、非线性线索与排列规则
━━━━━━━━━━━━━━━━━━━━

1. 线索类型包括：
direct：直接线索
feature：特征线索
negative：否定线索
relation：关系线索
cross：交叉线索
2. 线索强度严格区分：
direct 线索可以直接确认一个格子的答案，例如“对象01的某类别是某选项”。
negative 线索只能排除一个错误选项，不得在同一句中给出正确选项。
feature 线索只能描述可观察特征或间接特征，不得直接写出该格子的最终答案。
relation 线索只能说明两个或多个对象、类别或选项之间的关系，不得直接指出某对象的正确选项。
cross 线索必须通过两个或以上条件交叉推理得出结果，不得把交叉结果直接写在句子里。
凡是文字中已经明确写出“对象X对应某正确选项”的，都必须视为 direct 线索，不得标成 negative、feature、relation 或 cross。
3. 线索需要随机出现，不得按照上述类型固定顺序出现，不得按照对象顺序出现。

线索设计要求：
1. 初始线索设置2至3条，必须涉及至少两个档案对象和两个属性类别。初始线索应主要使用 direct；不得把 feature、relation、cross 或 negative 作为开局关键线索。
2. direct线索与后续出现的 feature、negative、relation、cross 等线索应尽量来自不同属性类别。
3. 不得让初始或后续线索全部集中在同一个对象、类别或type。
4. 后续线索应交错覆盖不同对象和属性类别。
5. 线索的解锁资格只能由 initial、unlockAfter、unlockMode 和已完成格子决定；不得因 type、JSON 数组位置或固定阶段强制解锁。
6. priority 和 JSON 顺序只可用于同批已解锁线索的稳定显示顺序，不得用于决定线索是否有资格解锁。
7. clues数组本身也应交错排列不同type和类别，不得把相同类型集中放置。
8. direct线索不得重复公布同一答案；relation和cross必须真正连接两个或以上属性类别。
9. 玩家应能从多个合理方向开始推理，不得只有单一固定作答路线。
10. 不同合理作答顺序均不得导致流程卡死。
11. 所有线索必须使用匿名代号，简短、准确、容易理解，并遵守【一A、全局通用规则】。
12. 每个对象必须在名称隐藏时达到66%，名称揭示后仍有线索完成剩余属性。
13. feature、relation、cross 属于间接或关联线索，不能设为 initial:true，必须设为 initial:false。它们只能在其中一条相关选项已被确认后才出现，因此 unlockAfter 必须至少填写一个相关格子ID，unlockMode 必须填写 "any"。
14. negative 属于否定线索，不能设为 initial:true，必须设为 initial:false。negative 只能在该类别只剩最后两个候选选项时出现：若该类别有 N 个对象/选项，则 unlockAfter 必须填写同一 category.id 下应先完成的 N-2 个格子ID，unlockMode 必须填写 "all"。
15. 间接线索禁止规则：
negative、feature、relation、cross 不得写成“不是A，而是B”“不属于A，而属于B”“排除A后就是B”等句式。
negative 只能排除错误选项，不能同时给出正确选项。
negative 线索数量不得超过2条。
relation 和 cross 只能提供关系或条件，不能在句子中直接写出最终答案。
如果一句线索已经直接写出某对象某类别的正确 option.label，该线索必须标为 direct，或重新改写为较弱的间接线索。

每条线索必须完整填写模板要求的：
id、text、type、initial、priority、relatedCells、completeAfter、completeMode、unlockAfter、unlockMode。

格子ID格式：
档案对象ID::类别ID

completeMode和unlockMode只能使用all或any。

若检查失败，必须调整分组、类别、线索或解锁条件，不得输出多解、死循环、身份泄露或无法完成的案件。
